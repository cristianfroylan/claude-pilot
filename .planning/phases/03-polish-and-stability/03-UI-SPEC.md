## Phase 3 Visual Delta

This section is the authoritative contract for every visual and layout change in Phase 3.

### 1. SafeArea Coverage

- **Applies to:** `TerminalScreen` Scaffold body
- **Contract:** Wrap the Scaffold body `Column` in a `SafeArea` covering **top only** (`SafeArea(top: true, bottom: false, left: false, right: false, child: ...)`)
- **Rationale:** The AppBar already handles top status-bar spacing on Android; on iOS with a notch, the AppBar does not extend under the notch automatically — `SafeArea(top: true)` prevents content from clipping under the notch. Bottom is excluded: the InputBar sits above the system navigation bar; `resizeToAvoidBottomInset: true` on the Scaffold (already set in Phase 1) handles the rest.
- **Verification:** On a physical iPhone with a notch (or the iOS simulator with a notch device), the terminal and InputBar must not render under the notch or home indicator at any orientation.

### 2. PermissionCard Text Overflow

- **Applies to:** `permission_card.dart` — the excerpt `Text` widget
- **Current state (Phase 2):** `overflow: TextOverflow.ellipsis` is specified in 02-UI-SPEC.md and in the component inventory code block. If the implementation omitted the property, this phase enforces it.
- **Contract:** The excerpt `Text` widget must carry `overflow: TextOverflow.ellipsis` and `maxLines: 1`. The 80-char truncation applied by the detector is the first guard; `TextOverflow.ellipsis` is the render-layer second guard.
- **No visual change from user perspective** if Phase 2 was implemented correctly — this is a correctness enforcement.

### 3. VoiceBottomSheet Scroll on Small Screens

- **Applies to:** `voice_bottom_sheet.dart` — the bottom sheet content column
- **Contract:** Wrap the internal `Column` in a `SingleChildScrollView` so that on devices with a screen height under ~600dp (e.g., iPhone SE, small Android phones), the sheet content does not overflow vertically.
- **Scroll axis:** vertical only
- **Physics:** `ClampingScrollPhysics()` — matches Material bottom sheet behavior; avoids bounce on Android
- **Layout structure after change:**

```
ModalBottomSheet (isScrollControlled: true)
  Padding(bottom: MediaQuery.of(context).viewInsets.bottom)
    SingleChildScrollView(physics: ClampingScrollPhysics())
      Column(mainAxisSize: MainAxisSize.min)
        [drag handle]
        [SizedBox h:16]
        [heading Text]
        [SizedBox h:12]
        [transcript Container]
        [SizedBox h:20]
        [Row: Discard | Send message]
        [SizedBox h:16]
```

- The sheet itself remains `isScrollControlled: true` (unchanged from Phase 2)
- On large screens, `mainAxisSize: MainAxisSize.min` keeps the sheet compact — `SingleChildScrollView` adds no visual change when content fits

### 4. PTY Resize — Keyboard and Rotation Robustness

- **Applies to:** `terminal_view_wrapper.dart`, `terminal_screen.dart`
- **This is a non-visual change** (no pixel-level design contract needed), but the interaction contract matters:
  - When the soft keyboard appears, the terminal **must reflow** within one frame after the inset settles — the user must not see stale column-width layout
  - When the device rotates, the terminal **must reflow** to the new column count — no clipped lines, no wrong-width diffs from Claude Code
- **Visual continuity contract:** During keyboard appear/disappear and rotation, the terminal background (`colorScheme.surface`, xterm TerminalTheme.background) must fill the full available area with no gap or white flash. The `TerminalView` must not show a blank frame between resize events.
- **Implementation note (from CONTEXT.md):** `LayoutBuilder` fires on widget rebuild. If keyboard-triggered inset changes do not propagate to the terminal's `LayoutBuilder` subtree reliably, a `MediaQuery`-aware listener on `viewInsets` must explicitly trigger a rebuild. The `addPostFrameCallback` pattern established in Phase 1 must be preserved.

### 5. SSH KeepAlive (Non-Visual)

- **Applies to:** `ssh_session_provider.dart` — `SSHClient` constructor
- **Contract:** `keepAliveInterval: const Duration(seconds: 30)` added to `SSHClient` constructor
- **No visual contract** — this has no UI rendering effect. It prevents silent TCP connection drops on iOS during backgrounding.
- **Interaction contract:** After the app is backgrounded for up to 30 seconds and brought back to foreground, the `TerminalView` must resume rendering new output without a manual reconnect. The connection status indicator must remain "Connected" (green dot) throughout if the keepalive succeeded.

No changes to accessibility contracts. Phase 3 changes do not alter semantic roles:

- `TextOverflow.ellipsis` on `PermissionCard` excerpt: the full text is passed to `permissionDetectorProvider` — the truncated display does not affect screen reader content (screen readers read the widget's `Text` data, not the rendered ellipsis)
- `SingleChildScrollView` in `VoiceBottomSheet`: the content order is unchanged; scroll does not affect traversal order
