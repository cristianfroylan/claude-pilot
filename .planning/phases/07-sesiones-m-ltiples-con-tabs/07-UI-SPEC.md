## Color

Same Material 3 dark ColorScheme from seed `#1E8BC3`. No new color values.

| Role | Material 3 Token | Approximate Hex | Usage in Phase 7 |
|------|-----------------|-----------------|------------------|
| Dominant (60%) | `colorScheme.surface` | ~#0F1117 | TerminalScreen body background; tab strip row background |
| Secondary (30%) | `colorScheme.surfaceContainerHigh` | ~#1A1D24 | Inactive tab chip background; AppBar background; machine selection drawer background |
| Active tab | `colorScheme.primaryContainer` | ~#1A3A4A | Background of the currently selected tab chip |
| Accent (10%) | `colorScheme.primary` | ~#4BA3C7 | Active tab label text; add session button icon; active tab close button icon |
| Disconnected error | `colorScheme.error` | ~#CF6679 | Error indicator dot on a failed/dropped tab |
| Connected dot | `Colors.green.shade400` | #66BB6A | Healthy connected session dot (matches Phase 1 convention) |
| Connecting dot | `colorScheme.secondary` | ~#4BA3C7 | Pulsing dot during connecting/reconnecting (matches Phase 4 convention) |

Accent (`colorScheme.primary`) reserved for ONLY these elements in Phase 7:
1. The label text of the currently active tab
2. The add-session button icon (`Icons.add`)
3. The close button icon on the currently active tab

Accent must NOT be used on: inactive tab labels, inactive tab close buttons, the tab strip background, the status dots (those use semantic colors above).

The active tab is distinguished from inactive by three simultaneous signals: background color (`colorScheme.primaryContainer`), label text color (`colorScheme.primary`), and a 2 dp bottom border line in `colorScheme.primary`. This satisfies WCAG 1.4.1 (color is not the sole indicator).

## Architecture Decision: Where the Tab Strip Lives

The tab strip is a **top-of-TerminalScreen persistent row** — not a Flutter `TabBar` widget and not `BottomNavigationBar`.

**Decision:** A custom horizontal `SingleChildScrollView` row pinned immediately below the AppBar in a new wrapper screen `SessionsScreen`. This wrapper replaces the direct `TerminalScreen` route. The router navigates to `/sessions`, which houses the tab strip and swaps the active `TerminalScreen` widget (each with its own keyed provider family entry).

**Rationale for custom strip over `TabBar`:**
- `TabBar` requires a `TabBarView` which creates/destroys tabs on scroll, destroying xterm PTY state — violating RECON-05 scrollback preservation.
- The custom approach keeps all `TerminalScreen` widgets in the widget tree simultaneously via `IndexedStack`, preserving every xterm buffer in memory regardless of which tab is visible.
- The strip scrolls horizontally when tabs overflow — Flutter `TabBar` has limited overflow control.

**Router change:** `/machines/:id/terminal` route is REMOVED. Machine list taps now navigate to `/sessions?newMachineId=:id`. If `/sessions` is already on the stack (sessions exist), a new tab is pushed into the tab strip instead of replacing the screen.

## Screen Inventory

### New Screen: Sessions Screen (`/sessions`)

This is the container that replaces the single `TerminalScreen` route. It always shows the tab strip and exactly one `TerminalScreen` (the active tab) at a time.

**Layout:** `Scaffold` with AppBar + tab strip row + `Expanded(IndexedStack)`.

| Element | Spec |
|---------|------|
| AppBar background | `colorScheme.surfaceContainerHigh` |
| AppBar title | Hidden when 1+ tab exists — the tab strip below provides context. If no tabs yet (edge case), show "Sessions" 16 sp semibold. |
| AppBar trailing action | `IconButton(Icons.add, tooltip: 'New session')` — opens machine selection sheet to pick which machine to open a new tab for |
| Tab strip row height | 44 dp |
| Tab strip background | `colorScheme.surface` |
| Tab strip layout | `SingleChildScrollView(scrollDirection: Axis.horizontal)` wrapping a `Row` of tab chips + one add button at the end |
| Tab chip (inactive) | `InputChip`-style container: `colorScheme.surfaceContainerHigh` background, 12 sp label in `colorScheme.onSurfaceVariant`, close icon `Icons.close` in `colorScheme.onSurfaceVariant`, height 36 dp (centered in 44 dp strip), `borderRadius: 8 dp` |
| Tab chip (active) | Same as inactive except: `colorScheme.primaryContainer` background, 12 sp label in `colorScheme.primary`, close icon in `colorScheme.primary`, 2 dp bottom border in `colorScheme.primary` |
| Tab chip minimum width | 80 dp; maximum width 140 dp (label truncates with `TextOverflow.ellipsis`) |
| Tab chip horizontal padding | sm (8 dp) left before status dot; xs (4 dp) between dot and label; xs (4 dp) between label and close button; sm (8 dp) right inside close button area |
| Status dot in tab | 8 dp diameter `Container(BoxDecoration(shape: BoxShape.circle))` — color follows session state (see Color section) |
| Close button in tab | `InkWell` or `GestureDetector` wrapping `Icon(Icons.close, size: 16)` — effective tap area padded to 32 dp × 32 dp |
| Add button (strip) | `IconButton(Icons.add, size: 20)` at the trailing end of the scroll row — `colorScheme.primary` icon — tooltip "New session" — 44 dp × 44 dp |
| Body | `IndexedStack(index: activeTabIndex, children: [TerminalScreen(machineId) for each tab])` — all TerminalScreen instances stay mounted |
| No-tab state | Never reached in normal flow (navigating to /sessions always opens a tab). If reached due to all tabs closed: show centered column "No active sessions" (16 sp semibold) + "Tap + to start a session" (14 sp, `onSurfaceVariant`) + `FloatingActionButton(Icons.add)` |

### Modified Screen: Machine List (`/machines`)

No structural change. The `onTap` handler for each machine tile changes target:
- Before Phase 7: `context.push('/machines/${machine.id}/terminal')`
- After Phase 7: `context.push('/sessions?newMachineId=${machine.id}')` — or equivalent programmatic tab open via `SessionsNotifier`

The machine list screen itself is visually unchanged.

### Removed Route: `/machines/:id/terminal`

This route is removed from the router. All terminal access goes through `/sessions`.

## Interaction Contract

### SESS-01: Opening a new session

1. User taps a machine in the Machine List → `/sessions` route receives `newMachineId`
2. `SessionsNotifier` creates a new tab entry `{id: uuid, machineId: machineId}` and appends it to the tab list
3. Active tab index switches to the new tab immediately
4. `IndexedStack` renders the new `TerminalScreen(machineId)` — SSH connection begins
5. Status dot on the new tab shows connecting pulse animation (`colorScheme.secondary`, 800ms opacity pulse, matching Phase 4 `_ConnectingDot` convention)
6. If `/sessions` is already active (user came from the add button): steps 2–5 only; no route navigation

Simultaneous tabs: All `TerminalScreen` widgets remain mounted in `IndexedStack` at all times. SSH sessions run independently — opening a new tab does not pause, disconnect, or clear any existing session.

### SESS-02: Switching tabs

1. User taps any inactive tab chip in the strip
2. `SessionsNotifier.setActiveTab(index)` updates active index
3. `IndexedStack` shows the corresponding `TerminalScreen` instantly (no animation, no reconnect)
4. Active tab chip visual state updates (background, label color, border — see Color section)
5. The previously visible terminal is not unmounted — its SSH session and xterm buffer are intact

No slide or fade animation between tabs. Instant switch is intentional: smooth animation on a terminal readability surface adds latency with no UX gain.

### SESS-03: Closing a tab

1. User taps `Icons.close` on any tab chip
2. If it is the only tab: full disconnect + navigate back to `/machines` (sessions screen becomes empty)
3. If multiple tabs remain: `sshSessionProvider(machineId).notifier.cancel()` is called to disconnect cleanly, tab entry is removed from the list, active index adjusts to adjacent tab (prefer left neighbor; if closed tab was leftmost, use new leftmost)
4. The closed tab's `TerminalScreen` is removed from `IndexedStack` — xterm buffer released
5. No confirmation dialog — closing a tab is recoverable by re-tapping the machine in the list

Tab close must NOT affect any other tab's SSH session state.

### SESS-04: Failed session indicator

When a tab's SSH session enters `SshFailed` state:

1. The status dot on that tab changes to `colorScheme.error` (#CF6679) — static, no pulse animation
2. The tab remains in the strip and retains its position
3. The terminal view within that tab shows the existing xterm scrollback buffer (RECON-05 preserved) plus the `ReconnectFailedOverlay` (already implemented in Phase 4)
4. No `SnackBar` is shown on the active tab for a *background* tab failure — the indicator on the tab chip is the only signal to avoid interrupting the currently active session
5. If the failed tab IS the active tab: existing Phase 4 `ReconnectFailedOverlay` behavior applies (SnackBar + retry button)
6. Tapping a failed tab to switch to it reveals the full `ReconnectFailedOverlay` with retry option — same Phase 4 behavior, no new UI needed

Other tabs: no visual change. Their SSH sessions are unaffected and their status dots reflect their own independent states.

### Tab strip scroll behavior

- When the combined width of all tab chips plus the add button exceeds the screen width, the `SingleChildScrollView` enables horizontal scroll
- No hard cap on maximum tabs — scroll handles overflow (soft expectation: typical user runs 2–4 sessions)
- When a new tab is added, the strip programmatically scrolls to make the new tab visible using a `ScrollController.animateTo` call
- When a tab is closed, no automatic scroll — existing position is preserved

### Back gesture / navigation

- Android back gesture while on `/sessions` with 1+ tab: does nothing (sessions are persistent — user must explicitly close tabs)
- This requires `PopScope(canPop: false, ...)` wrapper on `SessionsScreen`
- Closing the last tab navigates to `/machines` — machine list is always the root of the back stack

## State Management Contract

Phase 7 introduces a `SessionsNotifier` (Riverpod `Notifier<SessionsState>`, `keepAlive: true`) to manage the tab list independently of individual SSH session providers.

| Provider | Type | Purpose |
|----------|------|---------|
| `sessionsProvider` | `Notifier<SessionsState>` (keepAlive) | Owns the ordered list of `SessionTab(id, machineId)` entries and `activeIndex` |
| `sshSessionProvider(machineId)` | Existing family provider | Unchanged — each tab's SSH session lifecycle is managed exactly as before |

`SessionsState`:
- `tabs: List<SessionTab>` — ordered list of open tabs
- `activeIndex: int` — index of currently visible tab

`SessionTab`:
- `id: String` — UUID, unique per tab open event (allows same machineId to appear in multiple tabs)
- `machineId: String` — references the machine

The `sshSessionProvider` family key for a tab is `machineId`. If the same machine appears in two tabs, they share one SSH session provider entry. This is a deliberate simplification for Phase 7 — two tabs to the same machine show the same session. SESS-01 says "same or different machine" — same machine tabs are supported but share a session (not two independent sessions to the same machine). This decision avoids provider key collisions without introducing composite keys in Phase 7.

This constraint must be surfaced in the planner's task notes, not in user-visible copy.

## Accessibility

- Tab close buttons: `Semantics(label: 'Close [machineName] session', child: GestureDetector(...))` — machine name in label so screen readers distinguish multiple close buttons
- Status dot in tab: `Semantics(label: [connected|connecting|connection failed], child: Container(...))` — color alone is not the indicator
- Add session button: `IconButton(tooltip: 'New session', ...)` — Flutter renders tooltip as semantics label automatically
- Tab strip is a scroll area: the `SingleChildScrollView` scrolls horizontally — VoiceOver/TalkBack users can swipe through individual tab chips; no custom scroll semantics needed
- `IndexedStack`: only the active child is interactive to the user; inactive children are visually hidden but remain mounted. Wrap inactive `TerminalScreen` widgets in `Visibility(visible: false, maintainState: true, child: ExcludeSemantics(child: ...))` so accessibility tools do not expose multiple terminal outputs simultaneously
- `PopScope(canPop: false)` on `SessionsScreen`: does not affect screen reader navigation — only suppresses Android back gesture route pop
- Color + shape dual indicator on active tab: `colorScheme.primaryContainer` background + 2 dp bottom border + label color change — three simultaneous non-color signals ensure WCAG 1.4.1 compliance
