# Phase 04 Plan 02: Reconnection State Machine Summary

**One-liner:** Full reconnection state machine in SshSession — 5x initial backoff [1,2,4,8,16]s and 3x mid-session backoff [2,4,8]s with per-second Timer.periodic countdown, cancel/reconnect public methods, Terminal promoted to instance field, and all consumers migrated to AsyncValue<SshSessionState>.

## What Was Built

### Task 1: Reconnection State Machine in SshSession (commit 71bc8b1)

Refactored `lib/features/terminal/providers/ssh_session_provider.dart` from `AsyncValue<Terminal>` to a full reconnection state machine returning `AsyncValue<SshSessionState>`.

**Key structural changes:**

- `@Riverpod(retry: _noRetry)` with top-level `Duration? _noRetry(int, Object) => null;` function — correct Riverpod 3 idiom (static methods cause build_runner cast errors per GitHub #4332)
- `Terminal? _terminal` promoted from local variable in `_connectOnce()` to class instance field — created once via `_terminal ??= Terminal(maxLines: 2000)` in `build()`, reused across all reconnects (RECON-05)
- `build()` returns `Future<SshSessionState>` — runs a 5-attempt initial retry loop with backoff `[1, 2, 4, 8, 16]` seconds
- `_runMidSessionRetry()` — 3-attempt mid-session loop with backoff `[2, 4, 8]` seconds, triggered by `SSHClient.done`
- `_waitWithCountdown()` — `Timer.periodic` + `Completer<void>` pattern that emits state updates once per second and short-circuits on `_cancelRequested`/`_disposed`
- `_installDoneWatcher()` — extracted helper that captures `++_connectionGeneration` into a local `gen` to guard against stale done-callbacks from replaced `SSHClient` instances (Pitfall 5)
- `_stdoutSub` / `_stderrSub` — `StreamSubscription` fields cancelled at the start of each `_connectOnce()` call before re-subscribing (Pitfall 6)
- `cancel()` — sets `_cancelRequested = true` and stops `_countdownTimer` (RECON-03)
- `reconnect()` — single manual attempt, emits connecting state, reuses `_terminal`, does NOT call `ref.invalidateSelf()` (Pitfall 1 / RECON-04)
- `ref.onDispose()` remains the very first statement in `build()` before any awaits (Pitfall 4)

The `.g.dart` was regenerated by `dart run build_runner build` and now shows `$AsyncNotifierProvider<SshSession, SshSessionState>` with `retry: _noRetry`.

### Task 2: Consumer Migration (commit 3dd7eaf)

Three consumer files updated from `AsyncValue<Terminal>` to `AsyncValue<SshSessionState>`:

**`permission_detector_provider.dart`**
- Added import for `ssh_session_state.dart`
- Changed `data: (_) => permissionStream.map(_detect)` to a `switch` on `SshSessionState`: `SshConnecting()` returns `Stream.empty()`, all other variants (SshConnected, SshReconnecting, SshFailed) return the permission stream — rationale: all three carry an active terminal that may still surface permission prompts

**`input_bar.dart`**
- Added import for `ssh_session_state.dart`
- Changed `isConnected = ref.watch(...).hasValue` to `stateValue = ref.watch(...).value; isConnected = stateValue is SshConnected` — correct gate: input is only enabled when a live SSH session exists, not during reconnecting or failed states

**`terminal_screen.dart`**
- Removed `SshSession.maxAttempts` reference and `AlertDialog`/`showDialog` block
- `statusLabel` now uses `switch (sessionAsync.value)` on the sealed class: Connecting/Reconnecting/Failed/Connected/null
- `isPulsing` replaces `sessionAsync.isLoading`: covers `SshConnecting` and `SshReconnecting`
- Body `Expanded` child uses exhaustive switch: terminal-carrying variants render `TerminalViewWrapper`, `SshConnecting` and `null` render `CircularProgressIndicator`
- `ref.listen` shows lightweight `SnackBar('Could not connect to $machineName.')` on transition into `SshFailed` (Plan 04-03 replaces this with the full overlay)

### Architectural Notes

**`_installDoneWatcher()` extraction:** The plan specified inline `_client!.done.then(...)` with generation guard at three call sites (initial build, `_runMidSessionRetry()`, `reconnect()`). To avoid repeating the generation-guard logic three times with risk of inconsistency, the logic was extracted to a private `_installDoneWatcher()` method. This is a minor structural optimization, not a behavioral change. All three sites call `_installDoneWatcher()` and the generation guard works identically.

**`terminal_screen.dart` switch pattern:** The plan's `<action>` described a `switch` on `sessionAsync.value` with an `||` pattern combining `SshConnected`, `SshReconnecting`, and `SshFailed` in the body. Dart 3 supports `case X() || Y() || Z()` pattern guards with binding when ALL patterns bind the same variable — here `terminal` is a field common to all three variants, so `SshConnected(:final terminal) || SshReconnecting(:final terminal) || SshFailed(:final terminal)` compiles correctly and passes `terminal` to `TerminalViewWrapper`.
