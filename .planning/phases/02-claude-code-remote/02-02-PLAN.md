<objective>
Add the permission approval card: a sticky card that appears automatically above the InputBar
whenever Claude Code prints a permission prompt, with one-tap Approve (`y\n`) and Reject (`n\n`)
buttons that respond directly to the terminal.

The detection pipeline taps into the existing single stdout subscriber in `SshSession` via a
broadcast `StreamController`, feeds raw chunks to a new `@riverpod` StreamNotifier that applies
the `kPermissionPattern` regex, and drives an `AnimatedSwitcher` card slot in `TerminalScreen`.

This is a vertical slice: after this plan, a real permission prompt from Claude Code surfaces a
card the user can act on without typing.

Purpose: Deliver APRO-01, APRO-02, APRO-03 — approve/reject Claude Code actions with one tap.
Output: One modified provider, one modified screen, three new files (model, provider, card),
plus a build_runner run to generate the new provider's `.g.dart`. Also removes the leftover
debug `print` in ssh_session_provider.dart.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SSH stdout → regex detector | Untrusted remote terminal output is scanned for permission patterns |
| Card tap → PTY write | A tap writes a fixed `y\n`/`n\n` to the user's authenticated session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | Spoofing | Permission card from adversarial terminal output | accept | Per RESEARCH Security Domain: the SSH session is authenticated and LAN-only. Anyone who can write to the PTY already has SSH access. The card is a UX aid, not a security gate. |
| T-02-04 | Tampering | Echoed `y`/`n` re-triggering the card (dismiss-reappear loop) | mitigate | Approve/Reject call `ref.invalidate(permissionDetectorProvider(machineId))` immediately after `sendText`, dismissing the card on the same frame before the `y`/`n` echo can re-trigger it; the provider also emits `null` on the next non-matching chunk as a secondary clear (Pitfall 5). |
| T-02-05 | Information Disclosure | StreamController leak after session ends | mitigate | `_permissionController.close()` in `ref.onDispose` (Pitfall 3). |
| T-02-06 | Denial of Service | Regex applied to every stdout chunk | accept | `kPermissionPattern` is a bounded alternation over single lines; chunk sizes are small terminal writes. Acceptable for a single-user LAN tool. |
</threat_model>
